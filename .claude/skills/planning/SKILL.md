# planning（多步计划）

## 何时用

- 任务跨多个文件、多个步骤
- 改动有依赖关系（必须先做 A 才能做 B）
- 需要多人/多 agent 协作执行
- 改动有风险，需要明确验收点

**不要用于**：单文件小改动、纯问答类任务。

## 核心纪律

```
[Step N] → verify: [怎么验证这一步做对了]
```

每个步骤必须能独立验证（"我跑一下 X 命令就能确认"），不能是模糊的"看起来对了"。

## 我们项目的步骤

1. **拆解任务**：把目标拆成 3-10 个步骤。每步独立可验。
2. **写计划文档**：放项目 `docs/superpowers/plans/<topic>-plan.md`。
3. **决定执行模式**：单 agent 串行 / parallel-agents skill 并行 / 手动执行。
4. **执行**：每完成一步勾掉 verify check。
5. **完工后归档**：计划移到 `docs/superpowers/archive/`，更新 PROGRESS.md（如有）。
6. **PR 同步**：把计划内容贴进 PR 描述。

## 计划模板

```markdown
# <任务名> 实施计划

**目标**：一句话说清要达成什么

## 步骤

- [ ] Step 1: <做什么>
  - verify: <跑什么命令 / 看什么输出>

- [ ] Step 2: <做什么>
  - verify: <...>

## 不做什么

- 明确排除的范围
```

## 反模式

- ❌ 步骤不可验证（"修一下数据库连接"—— 怎么验证？）
- ❌ 步骤太大（"完成整个功能"—— 应该是 3-5 个子步骤）
- ❌ 不写 verify 就开干
- ❌ 计划写完不归档
