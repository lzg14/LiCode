# debugging（调试纪律）

## 何时用

- 用户报了一个 bug，但根因不明
- 性能回归，但不知道哪里变慢
- 间歇性失败、复现率低
- 多个可能根因，需要系统排查

**不要用于**：明显的拼写错误、单一文件的小 bug（直接修就行）。

## 核心纪律

```
复现 → 缩小 → 假设 → 仪表化 → 修复 → 回归测试
```

不要跳过"复现"直接猜，不要在没缩小范围前乱试修复方案，不要修复后不加 regression test。

## 我们项目的步骤

1. **复现**：用最小命令稳定触发，记录输入/环境/时间。
2. **缩小**：用 `git -C <path> bisect start` 找引入 commit；或 `git log --oneline -20` 缩小时间窗。
3. **已知问题**：用 `gh issue list --search "<keyword>"` 查 GitHub issues；`gh pr list --search "<keyword>"` 查 PR。
4. **隔离**：复杂调试起 worktree（用 git-worktrees skill），不污染主分支。
5. **仪表化**：Python 用 `python -X dev` 加 `pdb.set_trace()` 或 `icecream.ic()` 打印；其它语言用对应调试器。
6. **修复**：最小改动修根因，不顺手重构。
7. **回归测试**：加一个测试覆盖这个 bug，确保不再发生。

## 反模式

- ❌ 边猜边改、一次改三处然后跑测试看哪个好
- ❌ 在主分支乱加 print 调试
- ❌ 修复后口头说"应该好了" —— 必须有 regression test
- ❌ 不复现就开干
- ❌ 修改代码后立刻认为问题修复了（没跑测试验证）
- ❌ 在生产环境调试（应在本地或 staging）
- ❌ 忽略间歇性 bug 的"凑巧"复现（往往不是凑巧）
- ❌ 用 console.log 代替断点调试（复杂问题断点更高效）
- ❌ 把"修好了"当成已验证（必须有 regression test 证明）
- ❌ 只看报错信息不追踪调用栈（根因往往在更上层）
- ❌ 修复时绕过问题而不是解决根因（如 try-catch 吞掉异常）
- ❌ 不记录调试过程中的假设（复盘时无法回溯）
- ❌ 调试时改代码不记改了什么（事后无法复盘）
- ❌ 性能问题只看平均值不看 P99（尖刺才是用户体验杀手）
