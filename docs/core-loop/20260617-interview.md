**版本**: v1.8.0

## 1. Interview 机制（grill-me 风格）

### 1.1 核心原则

**参考 grill-me skill，内置为 Agent 默认行为。**

1. **一次只问一个问题** — 不要堆砌问题
2. **顺着设计树往下走** — 每个分支都要走到
3. **解决依赖关系** — 前置决策明确了再问后续
4. **每个问题给出推荐答案** — 帮助用户做选择
5. **能用代码探索回答的** — 不要问，自己查

### 1.2 Interview 流程

```
需求模糊 → 启动 Interview
    │
    ├── Q1: 用户类型？
    │       └── A1: 确认了
    │            │
    │            ├── Q2: 登录后跳转？
    │            │       └── A2: 确认了
    │            │            │
    │            │            └── Q3: 第三方登录？
    │            │                    └── ...
    │            │
    │            └── Q2': 权限模型？
    │                    └── ...
    │
    └── 所有分支走完 → 进入 ISA 编写
```

### 1.3 触发条件

| 等级 | Interview 强度 |
|------|---------------|
| E1/E2 | 不需要，影响范围明确 |
| E3 | 一句话确认："是这个意思吗？" |
| E4 | **强制 Interview**，每个分支走到 |
| E5 | **完整 Interview**，包括边界情况和失败模式 |

### 1.4 Anti-criteria 触发时机

**Anti-criteria 在 Interview 完成、所有分支理解清楚后统一展示。**

```
需求模糊 → 启动 Interview
    │
    ├── Q1: 用户类型？───► A1: 确认了
    │
    ├── Q2: 登录后跳转？───► A2: 确认了
    │
    ├── Q3: 第三方登录？───► A3: 确认了
    │
    └── 所有分支理解清楚
              │
              ▼
        Anti-criteria 统一展示
              │
              ▼
        用户确认弊端后 → 进入 ISA 编写
```

**触发规则**：
| 等级 | Anti-criteria 触发 |
|------|---------------------|
| E1/E2 | 不需要 |
| E3 | 可选，用户主动问"有什么风险"时展示 |
| E4 | **强制展示**，至少 3 个弊端 |
| E5 | **强制展示**，至少 5 个弊端 + 失败模式 |

### 1.5 Anti-criteria 数量要求

| 等级 | 最少弊端数 | 包括失败模式？ |
|------|-----------|----------------|
| E3 | 0（可选） | 否 |
| E4 | 3 | 是 |
| E5 | 5 | 是 |

### 1.7 外部知识搜索

**在需求理解后、进入 ISA 编写前，搜索是否有开源项目或已有实现。**

```
需求理解后 → 启动外部搜索
    │
    ├── 搜索开源项目：GitHub、GitLab、Gitee
    ├── 搜索已有实现：NPM、PyPI、Docker Hub
    └── 搜索文档：MDN、官方文档、Stack Overflow
              │
              ▼
        找到相似项目？
              │
         ┌────┴────┐
         │ 是        │ 否
         ▼          ▼
    提供给用户    继续原有流程
    比较建议
```

**搜索触发条件**：
| 等级 | 是否搜索 |
|------|----------|
| E1/E2 | 不需要 |
| E3 | 可选（用户同意时） |
| E4 | **强制搜索** |
| E5 | **强制搜索 + 详细对比** |

**搜索范围**：
| 类型 | 资源 |
|------|------|
| 开源项目 | GitHub, GitLab, Gitee |
| 包/库 | NPM, PyPI, crates.io, Maven |
| 容器镜像 | Docker Hub, ghcr.io |
| 解决方案 | Stack Overflow, Dev.to, 掘金 |

**输出示例**：
```
【外部知识搜索结果】
找到 3 个相似项目：
1. project-A (⭐2.3k) - 功能相似度 85%，不支持 XXX
2. project-B (⭐800) - 部分功能重叠，推荐作为参考
3. project-C (⭐100) - 功能差异较大，不推荐

建议：可直接使用 project-A 或基于 project-B 开发
```

### 1.8 核心原则

- **不理解不继续**：需求不清晰，强制 Interview，直到完全理解
- **做出来发现不是想要的 = 失败**：宁可多问，不要假设

## 3. 反向追问机制（Anti-criteria）

**不只是列出风险，是主动展示弊端，让用户充分了解再做决定。**

```
用户: "帮我加个这个功能"
    │
    ▼
系统识别：用户只考虑了正向需求
    │
    ▼
主动展示反向（弊端）：
┌─────────────────────────────────────────────────────────────┐
│ 【反向追问】这样做有什么弊端？                                  │
│                                                             │
│ 1. 性能影响：这个改动会增加多少复杂度？                         │
│ 2. 维护成本：后续维护难度会增加吗？                           │
│ 3. 耦合风险：会引入新的依赖吗？                               │
│ 4. 安全风险：有什么潜在的安全问题？                           │
│ 5. 迁移成本：现有数据/代码需要迁移吗？                        │
│ 6. 失败模式：如果失败了会怎样？                               │
│                                                             │
│ 请确认这些弊端你是否接受。                                    │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
用户确认后 → 继续
```

### 3.1 Anti-criteria 示例

| 场景 | 反向追问 | 弊端 |
|------|----------|------|
| 加缓存 | 缓存一致性问题？ | 数据可能过期、更新不生效 |
| 加日志 | 隐私合规？ | 可能记录敏感信息 |
| 加字段 | 废弃字段？ | 历史数据迁移成本 |
| 加依赖 | 供应链风险？ | 依赖项引入漏洞 |
| 删功能 | 破坏兼容？ | 已有调用方会报错 |

### 3.2 核心原则

- **不只问"要什么"，还要问"能接受什么弊端"**
- **用户不了解弊端 = 不完整的理解 = 不能动手**

## 4. 与 grill-me Skill 的关系

grill-me 是一个独立的 Skill，可以作为 Pai 的内置 Skill 之一调用。

但 **Interview 机制的核心逻辑是内置在 OBSERVE 和 THINK 阶段的**，不是作为外部 Skill。

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent Core Loop                          │
│                                                             │
│   OBSERVE ──→ 内置需求理解检查                               │
│       │                                                     │
│       └──→ 触发 Interview ──→ grill-me 逻辑（内置）         │
│                       │                                     │
│                       └──→ Anti-criteria ──→ 展示弊端        │
└─────────────────────────────────────────────────────────────┘
```

**调用关系**：
- Agent 内置 Interview 逻辑（核心）
- grill-me Skill 可作为参考实现
- 如果用户显式调用 `/grill-me`，使用 Skill 版本

## 5. 自动评审机制（Review Agent）

**核心思想**：主 Agent 做决策前，自动触发"反方"视角的评审，从逻辑漏洞、边界情况、替代方案等角度给出意见。

**适用范围**：评审需求、评审代码、评审方案、评审设计，覆盖多种场景。

```
主 Agent 提出方案
    │
    ▼
评审 Agent（自动触发）
    │
    ├── 逻辑漏洞
    ├── 边界情况
    ├── 替代方案
    └── 失败模式
          │
          ▼
    主 Agent 综合评审意见后再决策
```

### 5.1 与 Interview/Anti-criteria 的区别

| 机制 | 视角 | 目的 |
|------|------|------|
| **Interview** | 用户视角 | 理解用户真实需求 |
| **Anti-criteria** | 用户视角 | 确保用户了解弊端 |
| **Review Agent** | 反方视角 | 发现方案漏洞 |
| **E2E Review** | 第三方视角 | 全面评审 |

### 5.2 Review Agent 触发时机

| 场景 | 说明 |
|------|------|
| **评审需求** | E3+ 需求确认前 |
| **评审代码** | 代码变更前 |
| **评审方案** | 方案确定前 |
| **评审设计** | 架构变更前 |
| **自动评审** | 配置开启时，每次决策前 |

### 5.3 Review Agent 评审维度

| 维度 | 问题示例 |
|------|----------|
| **逻辑漏洞** | 假设是否成立？因果关系是否成立？ |
| **边界情况** | 空输入？极限输入？错误输入？ |
| **替代方案** | 有更好的实现方式吗？ |
| **失败模式** | 哪里会出错？失败了怎么恢复？ |
| **成本效益** | 投入产出比合理吗？ |

### 5.4 Review Agent 输出格式

```
【评审意见】
## 逻辑漏洞
- 第 3 步假设 X 成立，但实际可能不成立，因为...

## 边界情况
- 空输入时会崩溃，需要处理
- 极限输入时性能问题

## 替代方案
- 方案 B：更简单，缺点是...

## 失败模式
- 网络超时：需要重试 + 超时处理
- 依赖失败：需要 fallback

## 综合建议
建议采纳 / 建议修改后采纳 / 建议重新设计
```

### 5.5 Review Agent 模型配置

Review Agent 可以使用不同的 LLM/Model：

```yaml
review:
  # 评审使用独立 model
  provider: "anthropic"
  model: "claude-opus-4-20250514"  # 评审用更强的 model

  # 或使用专门的评审服务
  review_only_mode: true  # 不执行工具，只评审
```

**配置策略**：
| 策略 | 说明 |
|------|------|
| **同 provider 不同 model** | 与主 Agent 同 LLM，更便宜 |
| **不同 provider** | 用更强的 model 评审 |
| **本地 model** | 本地 Ollama 评审，更快更私密 |

### 5.6 与其他 LLM 评审的区别

| 评审方式 | 实现 | 说明 |
|----------|------|------|
| 你和我讨论 | 人类评审 | 设计阶段人工评审 |
| 其他 LLM 评审 | 离线文档评审 | 文档发给外部 LLM |
| **Review Agent** | **内置自动触发** | 每次决策前自动评审 |
