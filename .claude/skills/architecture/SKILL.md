# architecture（架构审视）

## 何时用

- 项目运行一段时间后想清理架构债
- 新成员加入前需要快速了解全局
- 改动前想理解模块边界
- 周期性的代码健康检查（如每个 sprint 一次）

**不要用于**：单文件审查（用 simplify）、单 bug 调试（用 debugging）。

## 核心纪律

```
扫 → 列 → 选 → 改
```

不要一次大爆炸改动多个模块；不要只列不改。

## 我们项目的步骤

1. **扫**：用 `cloc <project>` 看大文件、用 `radon cc -s <project>` 看高圈复杂度函数。
2. **列**：输出 markdown 清单：
   - 大文件（> 500 行）清单
   - 高复杂度函数（圈复杂度 > 10）清单
   - 循环依赖 / 紧耦合模块清单
   - "深模块"候选（小接口、宽功能）
3. **选**：和用户讨论，挑一个最有价值的改进点。
4. **改**：用 simplify + codebase-design skill 处理选中的点。
5. **回归**：跑 verification skill 全套检查。

## 输出模板

```markdown
# <项目名> 架构审视报告

**日期**：YYYY-MM-DD

## 大文件 Top 5

| 文件 | 行数 | 职责 |
|---|---:|----|

## 高复杂度函数 Top 5

| 函数 | 文件 | 圈复杂度 | 建议 |
|---|---|---:|----|

## 改进建议（按价值排序）

1. **<改进点>**：<为什么值得改>，<预估改动量>
2. ...
```

## 反模式

- ❌ 一次性改 5 个模块（大爆炸）
- ❌ 只列清单不行动
- ❌ 重命名 / 改 import 顺序当"架构改进"
- ❌ 不跑测试就重命名
