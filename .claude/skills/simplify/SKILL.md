# simplify（代码瘦身）

## 何时用

- 用户明确说"simplify"或"精简一下"
- review 时发现 over-engineering（自造抽象、不必要依赖、复杂设计）
- 代码 < 200 行能写完但写了一堆
- 性能没要求但代码太复杂

**不要用于**：性能关键路径、安全敏感代码（先确保对再瘦身）。

## 核心纪律

```
YAGNI → stdlib → 原生 → 依赖 → 一行 → 最小实现
```

不要做假设性抽象；不要引入未请求的依赖。

## 我们项目的步骤

1. **识别过度设计**：找自造的轮子、过度抽象、不必要的间接层。
2. **问"删了会怎样"**：能删就删。
3. **问"标准库能做吗"**：能用 `pathlib` 不用 `os.path`、能用 `dataclasses` 不用自己写 `__init__`。
4. **问"语言/平台原生能做吗"**：能用 `<input type="date">` 不引 datepicker 库。
5. **问"已装的依赖能做吗"**：能用现有依赖不引新的。
6. **砍未使用的 import / variable / 函数 / 文件**。
7. **跑测试 + ruff 确认没破**。

## 保留（不砍）

- **安全校验**：输入校验、权限检查
- **错误处理**：异常捕获、用户提示
- **可访问性**：a11y 属性
- **数据完整性**：事务、约束
- **回归测试**

这些不在瘦身范围内。

## 反模式

- ❌ 顺手把功能删了（瘦身 ≠ 砍功能）
- ❌ 引入新依赖代替旧代码（要看依赖是否真的需要）
- ❌ 把可读代码改成"聪明"的单行
- ❌ 瘦身后没跑测试
