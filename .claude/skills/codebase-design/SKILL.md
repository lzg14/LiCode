# codebase-design（深度模块设计）

## 何时用

- 设计新模块前
- 重构一个臃肿的类/函数
- 拆分过大文件时
- 决定模块边界时

**与 architecture 的分工**：architecture 看**全局**（多个模块的关系），codebase-design 看**单个模块**（一个模块的接口设计）。

## 核心纪律

```
宽功能 + 小接口 = 深模块
```

John Ousterhout（《A Philosophy of Software Design》）："好的模块是深的 —— 通过简单接口暴露大量功能"。

## 我们项目的步骤

1. **定义模块职责**：一句话说清这个模块负责什么。
2. **画接口**：只暴露必要的 public 方法/类，私有的标 `_` 前缀或放 `_internal.py`。
3. **藏实现细节**：把"如何做"藏在模块内部。
4. **设计测试**：测接口，不测实现细节。
5. **检查"接口大小 vs 功能大小"**：
   - 接口函数 < 5 个 → 可能太浅，考虑合并
   - 实现细节暴露在接口里 → 不深

## 设计检查清单

- [ ] 模块能用一句话说清职责
- [ ] 接口函数少（< 10 个）
- [ ] 调用方不用知道模块内部数据结构
- [ ] 内部状态不通过公共属性暴露
- [ ] 测试只通过 public 接口跑

## 反模式

- ❌ "薄模块"（接口函数多，每个功能少）—— 应当合并
- ❌ 把内部数据结构作为 public 属性（破坏封装）
- ❌ 一个模块做多件事（拆）
- ❌ 接口设计时考虑未来扩展（YAGNI —— 真需要时再加）
- ❌ 过早优化或过度工程化（简单问题简单解决）
- ❌ 用复杂的设计模式解决简单问题（KISS 原则）
