**版本**: v1.8.0

## 1. Interview 机制（grill-me 风格）

### 1.1 核心原则

**参考 grill-me skill，内置为 Agent 默认行为。**

1. **一次只问一个问题** — 不要堆砌问题
2. **顺着设计树往下走** — 每个分支都要走到
3. **解决依赖关系** — 前置决策明确了再问后续
4. **每个问题给出推荐答案** — 帮助用户做选择
5. **能用代码探索回答的** — 不要问，自己查

### 1.2 Interview 流程

```
需求模糊 → 启动 Interview
    │
    ├── Q1: 用户类型？
    │       └── A1: 确认了
    │            │
    │            ├── Q2: 登录后跳转？
    │            │       └── A2: 确认了
    │            │            │
    │            │            └── Q3: 第三方登录？
    │            │                    └── ...
    │            │
    │            └── Q2': 权限模型？
    │                    └── ...
    │
    └── 所有分支走完 → 进入 ISA 编写
```

### 1.3 触发条件

| 等级 | Interview 强度 |
|------|---------------|
| E1/E2 | 不需要，影响范围明确 |
| E3 | 一句话确认："是这个意思吗？" |
| E4 | **强制 Interview**，每个分支走到 |
| E5 | **完整 Interview**，包括边界情况和失败模式 |

### 1.4 Anti-criteria 触发时机

**Anti-criteria 在 Interview 完成、所有分支理解清楚后统一展示。**

```
需求模糊 → 启动 Interview
    │
    ├── Q1: 用户类型？───► A1: 确认了
    │
    ├── Q2: 登录后跳转？───► A2: 确认了
    │
    ├── Q3: 第三方登录？───► A3: 确认了
    │
    └── 所有分支理解清楚
              │
              ▼
        Anti-criteria 统一展示
              │
              ▼
        用户确认弊端后 → 进入 ISA 编写
```

**触发规则**：
| 等级 | Anti-criteria 触发 |
|------|---------------------|
| E1/E2 | 不需要 |
| E3 | 可选，用户主动问"有什么风险"时展示 |
| E4 | **强制展示**，至少 3 个弊端 |
| E5 | **强制展示**，至少 5 个弊端 + 失败模式 |

### 1.5 Anti-criteria 数量要求

| 等级 | 最少弊端数 | 包括失败模式？ |
|------|-----------|----------------|
| E3 | 0（可选） | 否 |
| E4 | 3 | 是 |
| E5 | 5 | 是 |

### 1.6 核心原则

- **不理解不继续**：需求不清晰，强制 Interview，直到完全理解
- **做出来发现不是想要的 = 失败**：宁可多问，不要假设

## 2. 反向追问机制（Anti-criteria）

**不只是列出风险，是主动展示弊端，让用户充分了解再做决定。**

```
用户: "帮我加个这个功能"
    │
    ▼
系统识别：用户只考虑了正向需求
    │
    ▼
主动展示反向（弊端）：
┌─────────────────────────────────────────────────────────────┐
│ 【反向追问】这样做有什么弊端？                                  │
│                                                             │
│ 1. 性能影响：这个改动会增加多少复杂度？                         │
│ 2. 维护成本：后续维护难度会增加吗？                           │
│ 3. 耦合风险：会引入新的依赖吗？                               │
│ 4. 安全风险：有什么潜在的安全问题？                           │
│ 5. 迁移成本：现有数据/代码需要迁移吗？                        │
│ 6. 失败模式：如果失败了会怎样？                               │
│                                                             │
│ 请确认这些弊端你是否接受。                                    │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
用户确认后 → 继续
```

### 2.1 Anti-criteria 示例

| 场景 | 反向追问 | 弊端 |
|------|----------|------|
| 加缓存 | 缓存一致性问题？ | 数据可能过期、更新不生效 |
| 加日志 | 隐私合规？ | 可能记录敏感信息 |
| 加字段 | 废弃字段？ | 历史数据迁移成本 |
| 加依赖 | 供应链风险？ | 依赖项引入漏洞 |
| 删功能 | 破坏兼容？ | 已有调用方会报错 |

### 2.2 核心原则

- **不只问"要什么"，还要问"能接受什么弊端"**
- **用户不了解弊端 = 不完整的理解 = 不能动手**

## 3. 与 grill-me Skill 的关系

grill-me 是一个独立的 Skill，可以作为 Pai 的内置 Skill 之一调用。

但 **Interview 机制的核心逻辑是内置在 OBSERVE 和 THINK 阶段的**，不是作为外部 Skill。

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent Core Loop                          │
│                                                             │
│   OBSERVE ──→ 内置需求理解检查                               │
│       │                                                     │
│       └──→ 触发 Interview ──→ grill-me 逻辑（内置）         │
│                       │                                     │
│                       └──→ Anti-criteria ──→ 展示弊端        │
└─────────────────────────────────────────────────────────────┘
```

**调用关系**：
- Agent 内置 Interview 逻辑（核心）
- grill-me Skill 可作为参考实现
- 如果用户显式调用 `/grill-me`，使用 Skill 版本
