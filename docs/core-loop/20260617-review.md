## 2. 待改进项

### 2.1 BUILD/EXECUTE 阶段边界模糊

**文件**: `seven-phase.md`

**现状**:
- BUILD："调用工具 + 准备决策"
- EXECUTE："产出实际输出 + 更新 ISA 进度"

**问题**: 两者的职责边界不清晰，实际实现中可能是原子操作。

**建议**: 明确 BUILD 是"执行工具调用"，EXECUTE 是"将工具结果转化为用户可交付物"，或合并为一个阶段。

### 2.2 Anti-criteria 触发时机不明确

**文件**: `interview.md`

**现状**: Anti-criteria 机制已定义，但与 Interview 流程的集成点不清晰。

**待澄清**:
- 是在 Interview 完成后统一展示所有 Anti-criteria？
- 还是每个分支追问时同步追问弊端？

**建议**: 明确 Anti-criteria 的触发点，例如"E4 Interview 完成后、ISA 编写前统一展示"。

### 2.3 权限继承机制未说明

**文件**: `multi-agent.md`

**现状**: §1.6 Fork Context 提到 `parentPermission`，但未说明子 Agent 的权限边界。

**待澄清**:
- 子 Agent 能否请求更高权限？
- 权限不足时是拒绝执行还是降级处理？

**建议**: 补充权限继承和升级机制说明。

### 2.4 Task 事件日志存储策略

**文件**: `multi-agent.md`

**现状**: §2.4 定义了 TaskEvent 数据结构，但未说明存储位置。

**待澄清**:
- 事件日志存储在内存中还是文件系统？
- 是否与 memory 系统整合？

**建议**: 明确事件日志的持久化策略。

### 2.5 Reasoning 摘要生成机制

**文件**: `context.md`

**现状**: §5.5 定义了摘要模板，但未说明摘要的生成方式。

**待澄清**:
- 摘要由主 Agent 生成还是子 Agent 生成？
- 生成摘要使用什么模型？

**建议**: 补充 Reasoning 摘要的生成机制说明。

## 3. 待澄清问题

### 3.1 E3 需求确认的定义

**矛盾点**:
- `effort-level.md` §2: E3 标记为"需求确认"
- `interview.md` §1.3: E3 只需要"一句话确认"

**待澄清**: E3 的需求确认强度到底是多少？

### 3.2 maxLifecycleAgents 上限

**文件**: `multi-agent.md` §1.5

**现状**: `maxLifecycleAgents: 1000`

**待澄清**: 长时间运行的 session 是否可能达到此上限？是否需要动态调整机制？

### 3.3 Topic 合并的消息顺序

**文件**: `context.md` §4.6

**现状**: `/merge topic1 topic2` 将两个话题的消息合并。

**待澄清**: 合并后消息按什么顺序排列？按时间戳排序还是保持各自顺序？
