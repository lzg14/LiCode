# domain-modeling（项目领域模型维护）

## 何时用

- 进入新项目，需要快速对齐术语
- 多个概念混用（agent 用了 20 个词其实是一个意思）
- 长期维护的项目（≥ 6 个月）
- 多 agent / 多人协作的项目

**不要用于**：临时脚本、一次性脚本、个人小项目。

## 核心纪律

```
建立共享语言 → 命名跟着走 → 文档化术语
```

mattpocock 原话："变量、函数和文件命名跟着共享语言走 → 代码库更易导航、agent 思考 token 更少"。

## 我们项目的步骤

1. **建 `CONTEXT.md`**：项目根建文件，结构：
   ```markdown
   # 项目术语表

   ## 核心概念

   - **<术语>**：<一句话定义>。不要和 <近义词> 混用。

   ## 业务规则

   - <业务规则 1>

   ## 缩写 / 别名映射

   | 缩写 | 完整术语 |
   |---|---|
   ```
2. **新功能前对齐**：开发前用 grilling skill 确认领域词汇。
3. **命名跟着走**：变量 / 函数 / 文件名都用 CONTEXT.md 里的术语，不用同义词。
4. **记录决策**：用 ADR（Architecture Decision Records）：
   ```markdown
   # ADR-001: <决策标题>

   **日期**：YYYY-MM-DD

   ## 背景
   <为什么需要做这个决策>

   ## 选项
   1. <选项 A>
   2. <选项 B>

   ## 决策
   <选了哪个>

   ## 后果
   <选了之后的影响>
   ```
   放项目 `docs/adr/0001-<topic>.md`。
5. **定期复审**：每个 sprint 复审 `CONTEXT.md`，加新术语、改定义。

## 反模式

- ❌ 术语表建了不更新（过期文档比没文档更糟）
- ❌ 同一概念用多个名字（"用户"、"客户"、"account"）
- ❌ 只在 README 写术语，不在 CONTEXT.md 单独维护
- ❌ 改了命名不更新文档
